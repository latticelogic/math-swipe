# i18n — languages, budgets, edge cases

Status (2026-07-16): infrastructure live (`src/i18n/`), Tier-1 UI shipped in
`en` + `es` + `pt-BR`. Owner's committed locale list includes zh-Hans,
zh-Hant, ja, hi (waves below). The language picker lands with the Settings
page; until then locale comes from browser detection (override:
`localStorage['math-swipe-locale']`).

## Which languages, and why this order

| Wave | Locales | Rationale | Gate |
|---|---|---|---|
| 1 (live) | en, es, pt-BR | Largest kids-app locales after English; Latin script → chalk fonts render unchanged; simple one/other plurals | — |
| 2 | fr, de | Big edu markets, Latin script (de strings run ~+35% long — budgets already account) | translation pass only |
| 3 | zh-Hans, zh-Hant, ja | Owner-committed. CJK breaks the chalk aesthetic (Fredericka/Architects Daughter are Latin-only) | **font pass**: per-locale `chalk` stack (candidates: Ma Shan Zheng zh, Yusei Magic/Klee One ja), subsetting pipeline (CJK fonts are MBs), halved char budgets |
| 3 | hi | Owner-committed. Devanagari — **Kalam** is a chalk-style font that covers it natively, so hi may land before zh/ja | font swap + translation |
| out | ar, he (RTL) | Layout mirroring interacts with swipe-direction answer mapping (left/right ARE answers) — a correctness project, not a translation | explicit product decision |

Singapore (home market) schools teach math in English → covered by `en`.

## Architecture (deliberate choices)

1. **Locale fixed per session; switching reloads.** No reactivity surface —
   `t()` is stable, memoized components and module-scope constants stay
   correct. Language switches are rare, deliberate actions.
2. **Tiny in-house module** (`src/i18n/`) — typed keys (`MsgKey`), `{var}`
   interpolation, CLDR plurals via `Intl.PluralRules`, English fallback for
   any gap. No i18next (~40KB) for 150 keys.
3. **Tiers.**
   - *Tier 1 (shipped)*: UI chrome — nav, rail, picker (groups + all 35
     category labels), end-run dialog, welcome/trial modals, paywall, share
     sheet, Me sign-in card + stat labels.
   - *Tier 2 (English until native review)*: teacher voice pools, chalk
     messages, achievements names/descs, WeeklyRecap lines, share payload
     text. These carry the app's voice — machine translation would break the
     tone bar. The pipeline: native-speaker pass per locale, then move keys
     into the catalogs.
   - *Tier 3*: the 36 Magic Tricks lessons (largest corpus, pedagogy +
     wordplay). Per-locale effort ≈ writing a small book; scope per market
     demand. A locale without trick translations shows tricks in English.
4. **Never concatenate fragments** — full-sentence templates with named
   placeholders only ({price}, {days}, {label}), so word order can differ.

## Max char lengths (enforced by test)

`i18n.test.ts` fails CI if any translation overflows its slot. Latin-script
budgets (CJK will use ~half):

| Slot | Budget | Why |
|---|---|---|
| Bottom-nav pill | 14 | 4 pills share 430px |
| Rail tooltip / toast | 34 | single line, floats left of the rail |
| Picker tile label | 14 | 3-column grid, 90px tiles |
| Picker section header | 20 | uppercase + tracking-widest |
| Picker chrome | 24 | sub-headers, back link |
| Primary buttons | 28 | full-width CTA, one line |
| Trial chip | 48 | always-visible pill in Me |
| Stat labels | 12 | 4-up stat row |
| Share tiles | 16 | 5-column grid |

Budgets are measured with placeholders worst-case-expanded. The test also
enforces **key parity** (es/pt-BR must cover exactly the en key set),
**placeholder parity** (a renamed {var} fails), **plural completeness**
(.one/.other per plural base), and the **no-emoji rule across all catalogs**.

## Edge cases (handled or explicitly decided)

- **Plurals**: `Intl.PluralRules` per locale. en/es/pt one/other; zh/ja are
  'other'-only; hi counts 0..1 as 'one'. `tCount()` handles all of these
  today — CJK/hi catalogs need no code change.
- **Decimal separators — THE math-app trap**: es/pt/de write 3,14; our
  problems/options write 3.14. **v1 policy: math notation is locale-invariant
  ('.' decimals, en-US grouping)** because generators compare answers by
  value and the notation is part of the taught material. Revisit with native
  reviewers per market (Brazilian schools teach comma-decimals — this is the
  first thing a pt-BR reviewer will flag). UI counts (XP totals) DO use
  locale formatting via `nf()`.
- **Currency**: the $3.14 price is USD on every channel (Airwallex + Play
  both charge USD) and renders as "$3.14" untranslated. Localized *pricing*
  (Play per-country prices) is a Play Console feature — if ever enabled, the
  price label must come from Play's Digital Goods `getDetails()` instead of
  the constant.
- **Dates**: `shortDateLabel()` (Intl) — "Jul 16" / "16 jul". No manual
  formats anywhere.
- **Fonts**: wave-3 gate above. Chalk headings in non-covered scripts fall
  back to system fonts (functional, off-aesthetic) — hence the gate.
- **Truncation**: CSS ellipsis only (grapheme-safe in browsers). Never
  `slice()` user-visible text — Devanagari conjuncts and emoji break.
- **Legal pages**: English only, on purpose. They're the governing text;
  translations need counsel per locale. The footer label row may translate
  later, the bodies do not.
- **Push notifications**: server-side copy stays English v1. Per-locale
  push needs the client to store its locale in `pushSubscriptions` — noted
  in functions TODO when demand exists.
- **SEO / hreflang**: no locale routing in v1 (client-side locale, one URL).
  If store data shows es/pt traction, add `/es/` prerendered landing +
  hreflang then.
- **Share payloads**: sender's locale (recipient unknown). The 🟩 grid is
  language-neutral by design.
- **aria-labels**: translated alongside visible text (same catalogs).
- **Missing keys**: English fallback + parity test means the only way to
  leak English is to add a key and fail CI.

## Adding a locale (the checklist)

1. Add the id to `Locale`; add `<locale>.ts` catalog (copy en.ts, translate).
2. Parity/budget/emoji tests pass → wired automatically (`CATALOGS` map +
   `SHIPPED_LOCALES` entry + detection mapping in `matchShipped`).
3. Non-Latin script? Font stack first (wave-3 gate).
4. Native-speaker review for tone (Tier 1 minimum) before marketing in that
   language.
