# Math Swipe — Mobile Audit, 2026-05-11

**Scope**: Comprehensive walkthrough of the production app at https://math-swipe-c7k.pages.dev across four mobile-class viewports (360×800, 412×915, 768×1024) and both color themes. Goal: surface every polish issue between current state and commercial-launch quality, in one batch, so the next fix-PR can be focused.

**Method**: Playwright-driven walkthrough of every tab, both bands (Starter/Full), both themes, with screenshots captured throughout. ~25 screenshots, ~50 minutes elapsed.

**Headline finding**: The app's **content and architecture are excellent**. The chalkboard aesthetic, hand-drawn icons, generated math problems, achievement system, magic-tricks lessons, and league all feel like a genuine product. The gap between current state and launch-readiness is **smaller than feared** — almost entirely in the categories of (a) font reliability, (b) tap-target sizes, (c) a few specific layout overflows, and (d) emoji-vs-SVG icon consistency. None of the findings indicate architectural problems.

## Triaged findings

### P0 — Blockers

*(None found.)* The app deploys cleanly, all core flows work, no crashes encountered.

### P1 — Must fix before launch

#### P1-A. Custom fonts load fragilely (intermittent serif-fallback on first load)

**What**: Both branded fonts (Fredericka the Great, Architects Daughter) trigger `net::ERR_FAILED` on the first load attempt from `fonts.gstatic.com`. They retry and usually recover within the same page load on a fast desktop browser, but on slower mobile networks the first impression can be the serif fallback. This is the tester's "the beautiful fonts that made the game distinctive are gone" complaint.

**Why**: The CSP in `public/_headers` allows `fonts.gstatic.com` under `font-src` but NOT under `connect-src`. Modern Chromium has tightened to require both for font fetches via the FontFace API. The retry succeeds because Chromium takes a different code path that satisfies the looser `font-src` rule alone.

**Verified**:
- `document.fonts` after page load shows both fonts in *both* `unloaded` AND `loaded` states (the first attempt failed; a later attempt succeeded).
- `new FontFace(url).load()` directly → `NetworkError`.
- `curl -sI <font-url>` → 200 OK (Google serves them; browser blocks).
- The font *does* eventually apply: `getComputedStyle(score).fontFamily === "Fredericka the Great", cursive`.

**Recommended fix** (in priority order):

1. **Self-host the two woff2 files.** Copy them into `public/fonts/`, update `index.css` `@font-face` to reference local paths. Eliminates the network failure mode, the CSP complexity, the extra DNS lookup, and a known GDPR risk (German courts have fined sites for sending IPs to Google via the Fonts CDN). This is the right long-term move.
2. **Band-aid: add `https://fonts.gstatic.com` to `connect-src`** in `public/_headers`. Reduces but doesn't eliminate first-load fragility.

#### P1-B. Lesson Mode equation overflows on final step

**What**: PR #26 fixed the lesson-mode equation overflow for the intro state (`34 × 12 = 34 × 10 + 34 × 2 = ?`), but the *final* step appends the answer (`= 408`) and the now-longer equation clips on the right. Visible on both 412×915 and 360×800 viewports. Result: user sees "= 4..." with the actual answer truncated.

**Why**: The CSS `font-size: clamp()` formula was tuned to fit the intro-state width, not the final-state width. The clamp doesn't re-evaluate per-step content.

**Recommended fix**:
- Quickest: tighten the clamp from `5vw` to `4vw` (or `3.5vw`) so even the longer final-state fits at 360px width.
- Better: replace clamp with the JS-based scale-to-fit approach (`AutoFitEquation` from my earlier failed attempt) but make it work by waiting on `document.fonts.ready` + a one-frame `requestAnimationFrame` before measuring.

**Impact**: Magic lessons are the *educational* core of the app. Truncated answers undermine the whole teaching premise.

#### P1-C. In-game share button click silently no-ops on the home screen

**What**: Tapping the share icon (top-right action sidebar) on the home screen — before any question has been answered — produces no visible response. No modal, no toast, no console error.

**Why** (suspected): The share button at `ActionButtons.tsx` calls `navigator.share` with the current game URL. If `navigator.share` is unavailable or rejects, the fallback is `navigator.clipboard.writeText()` with no toast. The tester taps and thinks the button is broken.

**Recommended fix**: Add a small toast or visual feedback for both the success path ("Copied link!") and the silent-fallback path. The current code already has `console.error` logging (from PR #23) but that's invisible to users.

**Note**: This is a different code path than the post-session SessionSummary share — the SessionSummary path is in better shape (has a proper modal). The in-game share is the rougher one.

### P2 — Should fix before launch

#### P2-A. Top-right controls miss tap-target minimum

**What**: Two buttons in the top-right corner are under the 44pt iOS / 48dp Android tap-target minimum:
- "Change level" (band switcher): 53×24 px — only 24 high
- "Toggle theme" (sun/moon): 36×36 px

**Why it matters**: Both sit in the hardest thumb-reach area. Small targets here are doubly punishing. Older users and kids developing fine motor control (our Starter band audience) will mis-tap.

**Fix**: Bump padding. `px-2 py-1` on Change level → `px-3 py-2.5`. Toggle theme button needs `p-1` more padding. Targets should be ≥44×44px.

#### P2-B. Emoji icons clash with the hand-drawn aesthetic in multiple surfaces

**What**: Several screens use OS emoji icons that render in the system's color emoji font, breaking the otherwise-consistent chalkboard / hand-drawn style:
- Question-type picker modal: Doubles 👯, Skip Count 🦘, Compare ⚖️, Shapes 🔺, Bonds 🔗, Skip Count, etc.
- League page: crown 👑, lightning ⚡, fire 🔥 next to player names.
- Magic School trick rows: 🖐, ➿, 🚀, 🕛, ⏱, 🪙, ⏬, 🪞, etc.

**Why it matters**: The band switcher (sapling/rocket SVGs) and the Me page achievement badges show what consistent, on-brand iconography looks like. The emoji-using surfaces look unfinished by comparison. Direct user feedback was "we like SVGs, not emojis."

**Fix**: Replace category, league, and trick icons with stroke-style SVGs matching the band-switcher style. Single-character chalk-text icons (`+`, `−`, `½`, `10`) can stay if they fit the style.

**Estimated effort**: ~30 SVGs × ~5-10 minutes each = 3-5 hours of icon work. Could be partly outsourced.

#### P2-C. Locked achievement badges have poor contrast

**What**: The achievement grid on the Me page shows locked badges in low-contrast grey outlines on the dark background. Users have to peer to read achievement names. Counts as "I can't see what I'm unlocking" — directly undermines the achievement-driven retention mechanic.

**Fix**: Bump locked-badge outline opacity from probable `~10%` to `~25%`. Or use a faint inner fill in addition to the outline so the locked badge has a silhouette readable at a glance.

#### P2-D. "Daily challenge available" pill is barely visible

**What**: When the daily challenge is available, a small pill appears under "Let's Go!!" reading "📅 Daily challenge available". On both light and dark themes the text is washed out — easy to miss as a returning user.

**Fix**: Either make the daily challenge pill more prominent (chalk gold, larger) since it's a critical retention hook, OR move daily challenge entry to a more obvious place (a banner, or first card under the score).

### P3 — Polish backlog

#### P3-A. Question-type picker scrollbar styling (mostly already fixed)

The picker modal scrolls — the global themed-scrollbar styling I shipped in PR #26 already handles this. Verified clean on mobile viewports (scrollbar hidden via `@media (pointer: coarse)`).

#### P3-B. Teacher character overlaps action sidebar

The teacher in the bottom-right extends up high enough to share pixels with the bottom of the action sidebar (skull icon). Not a blocker — still functional.

**Fix**: Lower teacher max-height by ~10% OR add a soft mask gradient near the sidebar OR reposition.

#### P3-C. "Let's Go!!" greeting persists indefinitely until first answer

The greeting is striking on first launch but slightly empty on every revisit. Designer call whether to keep the consistent identity or transition to something quieter after first interaction.

#### P3-D. Speedrun tab — selected-tab pill has unusual contrast

The selected "Speedrun" tab is purple-on-teal-border. Looks vivid but slightly off-tone with the rest of the chalkboard palette.

#### P3-E. Chalk color swatches row layout

The chalk color row in Me page customization wraps in a slightly awkward way at 412px (two rows but the second row is shorter than expected). Probably a CSS gap or wrapping issue — minor.

#### P3-F. League page is sparse (3 fake entries + "you")

With only 3 leaderboard entries on production, the league feels empty. This is a content/data problem, not a code problem — the page itself is well-built.

**Fix**: Seed more synthetic competitors at launch, or surface the user's own historical sessions as additional entries until real users arrive.

#### P3-G. Daily Challenge mode lacks a prominent in-progress indicator

When playing the daily challenge, only the small calendar icon in the action sidebar indicates the mode. Some users may not realize they're in a special session.

**Fix**: Add a "DAILY CHALLENGE · 3/10" banner at the top, similar to how Lesson Mode and Practice Blitz show their headers.

#### P3-H. Tablet viewport unused — black bars on either side

At 768×1024, the app correctly stays mobile-shaped with ~430px width and black space either side. Functionally fine; could be a missed opportunity to deliver a richer tablet experience (e.g., show stats / history alongside the game).

**Fix**: Defer. Tablet-optimized layout is real work and not a launch blocker. Mobile-only is a defensible choice for v1.

## Items NOT inspected (audit limitations)

- **Post-session SessionSummary share modal**: confirmed exists in code but could not trigger through automation (the swipe-gesture input model resisted my dispatched click events). Manual testing required.
- **Offline behavior**: not tested. App is PWA-shaped with a service worker; offline experience could be verified by toggling network throttling.
- **Push notifications**: not exercised. The Me page shows "Turn on notifications" button — unclear what the post-permission state looks like.
- **Account claim flow**: started anonymously, didn't go through Google / Email sign-in.
- **Light theme on tabs other than Game**: only tested home screen in light mode.
- **Accessibility (screen reader)**: not tested. Worth a pass with a real screen reader before launch.

---

## Strengths worth preserving

This audit is mostly about gaps, but the audit also surfaces what's working:

1. **The Magic School page is exceptional.** The presentation of mental-math tricks with the gold "recommended for you" callout, organized sections, and per-trick level pills is a polished product. This is a real differentiator.
2. **Hand-drawn teacher characters and achievement badges** are charming and consistent. Wherever the app commits to SVGs over emojis, it looks great.
3. **The chalkboard aesthetic** — when the fonts load — is distinctive and reads as a real brand. Don't lose this.
4. **The auto-difficulty system** (1-5 slider that adjusts on response time) is a sophisticated UX choice that hides complexity from users. Good design.
5. **Two-band Starter/Full** simplification feels right. Adults won't get patronized; kids get a real curriculum.

## Recommended action plan

**Phase A (one PR, ~1-2 days of focused work) — Launch blockers**:
1. Self-host the two custom fonts (P1-A).
2. Fix Lesson Mode final-step equation overflow (P1-B).
3. Make in-game share button give visible feedback (P1-C).
4. Bump tap targets on Change level + Toggle theme (P2-A).

**Phase B (one PR, ~1-2 days) — Visual consistency**:
5. Replace remaining emoji icons with SVGs in the question-type picker, league, magic tricks (P2-B).
6. Improve locked-achievement contrast (P2-C).
7. Make daily-challenge pill more discoverable (P2-D).

**Phase C (defer) — Polish backlog**:
- Everything in P3, prioritized by tester feedback after Phase A+B ships.

If Phase A and B both ship, the app crosses the bar from "rough" to "presentable." If only Phase A ships, the tester's specific complaints resolve but the emoji-mix issue remains visible.

**My recommendation**: ship both phases before any wider testing or store submission. Two focused PRs is faster than five small ones, and the audit work is done — the fixes themselves are mechanical.
